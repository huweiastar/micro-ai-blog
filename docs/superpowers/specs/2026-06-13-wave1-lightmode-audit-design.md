# Wave 1：亮色模式全面验证与修复 — 设计文档

**日期**：2026-06-13  
**目标**：系统验证亮色模式在全站所有页面的显示效果，识别并修复样式不一致问题，完成后直接部署上线。

---

## 概述

项目已实现亮色模式支持（`next-themes`），但缺乏系统的视觉验证。本 Wave 通过逐页面浏览器检查，确保亮色模式与深色模式具有相同的视觉质量和易读性，然后修复发现的问题。

---

## 验证范围

### 1. 首页（`app/page.tsx`）
- **元素**：导航栏、Hero 区、专栏卡、统计面板、热门文章、项目展示、页脚
- **关键点**：
  - 背景与文字对比度（WCAG AA 标准）
  - 卡片边框、阴影、背景色
  - 渐变和装饰性元素的可见性
  - 分类色彩的亮度适配

### 2. 文章列表（`app/blog/page.tsx`）
- **元素**：BlogCard、分类徽标、标签、筛选器
- **关键点**：
  - 卡片背景与文字的对比度
  - 分类色「书脊」的可见性
  - 悬停（hover）效果是否清晰
  - 阅读时长、发布日期等元信息的颜色

### 3. 文章详情页（`app/blog/[slug]/page.tsx`）
- **元素**：文章标题、正文、代码块、Callout、图片、TOC、评论区、系列导航
- **关键点**：
  - **代码块**：背景色、文本颜色、高亮符号的清晰度
  - **Callout（NOTE/TIP/WARNING/IMPORTANT/CAUTION）**：边框色、背景色、图标的亮度
  - **正文**：行间距、字体颜色、链接颜色与下划线
  - **图片**：边框、阴影（特别是浅色背景的图片）
  - **TOC 侧栏**：背景、文字、活动链接的对比度
  - **评论区（Giscus）**：评论背景、文字的亮度

### 4. 分类/标签/归档页
- `app/categories/page.tsx` — 分类列表、分类详情
- `app/tags/page.tsx` — 标签列表
- `app/archive/page.tsx` — 按年月分组的文章列表
- **关键点**：列表卡片、分组标题、计数徽标的颜色一致性

### 5. 其他页面
- `app/about/page.tsx` — 个人简介、技术栈、社交链接
- `app/projects/page.tsx` — 项目卡片、描述、技术标签
- `app/search/page.tsx` — 搜索结果、高亮关键词
- `app/footprint/page.tsx` — 如有

### 6. 后台（`app/admin/*`）
- **页面**：
  - `/admin` — 仪表盘（卡片、图表、统计）
  - `/admin/articles/edit` — 编辑器、检视器、预览
  - `/admin/projects/edit` — 项目编辑器
  - `/admin/categories` — 分类编辑
  - `/admin/theme` — 主题设置、媒体库
  - `/admin/content-health` — 内容体检结果
- **关键点**：
  - 表单输入框的边框、背景、占位符文字
  - 按钮的背景、文字、禁用态
  - 模态框、抽屉的背景与边界
  - 编辑器界面的工具栏、菜单
  - 检视器卡片的背景与文字

### 7. 移动端（手机/平板）
- 上述所有页面在 `sm`（640px）、`md`（768px）、`lg`（1024px）断点下的响应式显示
- **关键点**：
  - 导航菜单的可读性（特别是侧边栏/抽屉）
  - 卡片在窄屏下的排版
  - 按钮、输入框的触控大小（min-h-11/min-w-11）

### 8. 动画与过渡
- **View Transitions**：页面切换时的 `page-enter` 动画在亮色模式下的视觉效果
- **Hover 效果**：按钮、链接、卡片的悬停反馈
- **减速动画**：`prefers-reduced-motion` 时的降级表现

## 检查清单（验证方式）

### 阶段 1：环境准备
- [ ] 启动开发服务器：`NEXT_DIST_DIR=.next.dev PORT=3001 npm run dev`
- [ ] 确认服务器运行在 `http://localhost:3001`
- [ ] 清空浏览器缓存（F12 → Application → Clear storage）

### 阶段 2：逐页面系统检查
对每个页面，分别在**深色模式和亮色模式**下进行以下检查：

1. **对比度检查**
   - 使用浏览器 DevTools 的 Lighthouse 或专用工具（如 WAVE）检查 WCAG AA 标准合规性
   - 重点关注：文本与背景、边框与背景

2. **视觉一致性检查**
   - 颜色：是否有色彩在亮色模式下过浅或过深
   - 边框：亮色模式下是否可见
   - 阴影：是否适配亮色背景
   - 渐变：是否清晰可见

3. **功能检查**
   - 交互元素（按钮、链接）是否可点击且可识别
   - 表单输入框是否清晰
   - 代码块内容是否可读

### 阶段 3：问题记录
记录格式：
```
页面: [URL]
模式: 亮色 / 深色 / 两者都有
问题描述: [具体现象]
位置: [涉及的组件/文件]
严重度: P0（阻塞）/ P1（视觉明显） / P2（细节）
```

### 阶段 4：修复策略
根据问题严重度分类处理：

**P0（阻塞）**：无法读取内容、严重对比度不足
- 立即修复

**P1（视觉明显）**：色彩不协调、边框/阴影不可见
- 优先修复

**P2（细节）**：细微色差、细节调整
- 如时间允许，修复；否则记录为后续优化

## 实施步骤

### Step 1：启动环境
```bash
NEXT_DIST_DIR=.next.dev PORT=3001 npm run dev
```

### Step 2：逐页面检查与记录
按验证范围顺序，用浏览器同时打开两个标签页（深色 + 亮色），逐一检查并记录问题。

### Step 3：修复
根据记录的问题，在以下文件中修改样式：
- `app/globals.css` — 全局样式、CSS 变量、theme 相关
- `config/theme.json` — 主题配置（如需）
- 各组件的 `dark:` 前缀逻辑 — 确保亮色模式的对应样式完整

修复方式：
- 若 `dark:` 已有对应亮色样式，检查值是否合适
- 若缺少亮色对应样式，补充 light mode 样式或调整 CSS 变量

### Step 4：验证修复
逐一验证修复后的页面，确保问题已解决。

### Step 5：清理与提交
```bash
# 清理开发产物
rm -rf .next.dev

# 恢复可能被修改的文件
git checkout tsconfig.json public/{sitemap.xml,rss.xml,knowledge-index.json,search-index.json}

# 提交修复
git add [修改的文件]
git commit -m "feat(theme): 亮色模式全面审视与修复

- [具体修复项列表]"
```

### Step 6：部署
```bash
./deploy.sh
```

## 关键约束

1. **不修改功能逻辑** — 仅涉及样式修复
2. **保留深色模式质量** — 修复过程中不降低深色模式的显示效果
3. **尽量复用 CSS 变量** — 避免硬编码颜色值，优先通过 CSS 变量系统实现
4. **测试 prefers-color-scheme** — 确保首访时自动适配系统偏好
